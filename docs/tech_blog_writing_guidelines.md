# テックブログ執筆ガイドライン：開発と並走するために

「実験が終わってから書く」のではなく、**「実験・実装しながら書く」** ことを推奨します。
後から思い出して書こうとすると、当時の「課題感」や「試行錯誤の熱量」が失われ、ありきたりな記事になりがちです。

本ガイドラインでは、開発と執筆をスムーズに統合するためのルールをまとめます。

---

## 1. 鉄則：同時進行の法則 (Experiment & Write)
- **ルール**: 実装や実験を行うのと**同時に**、ブログ（またはメモ）にその内容を記録すること。
- **Why**:
  - 時間が経つと、「なぜそのアプローチを取ったのか」「何に困っていたのか」という鮮度の高い**問題意識（Problem Awareness）**を忘れてしまう。
  - 読者が一番共感するのは、成功した結果よりも、その過程で抱いた「悩み」や「課題感」であるため。

## 2. 再現性の担保：コマンドを残す
- **ルール**: 実行したコマンドは、成功・失敗に関わらず**その場で**記録する。
- **Action**:
  - `history` コマンドやターミナルのログをコピペしておく。
  - 引数や設定ファイル (`config.yaml`) のどのバージョンを使ったかを紐付けておく。
- **Why**:
  - 「あの時の実験、どうやって動かしたっけ？」と再現できなくなるのを防ぐ（自分自身のための備忘録）。
  - ブログの「実行コマンド」セクションの信頼性を高める。

## 3. アセット管理：こまめな `git add`
- **ルール**: 参照するスクリプト、ログファイル、生成されたグラフ・画像は、実験の一環としてこまめに `git add / commit` する。
- **Action**:
  - 実験用のスクリプトだからといって `tmp/` などに放置せず、`experiment/scripts/` などに整理してバージョン管理する。
  - ログデータや画像も、重要なものはリポジトリ（または外部ストレージへのリンク）として永続化する。
- **Why**:
  - ブログ執筆時に「あの時のグラフ画像どこいった？」となるのを防ぐ。
  - コードと結果（ログ）がセットで管理されていることで、読者が追体験しやすくなる。

## 4. 失敗の記録：供養の精神
- **ルール**: 失敗した実験も、**「なぜ失敗したかの考察」**と共に記録する。
- **Why**:
  - うまくいかなかったパターン（Negative Results）は、世の中に出回りにくいため、逆に価値が高い。
  - 「Aを試したがダメだった → だからBを試した」というストーリーは、結論（Bが正解）単体よりも説得力がある。

## 5. 実装＝執筆 (Development-Driven Documentation)
- **ルール**: PRのDescriptionやコミットメッセージを書くつもりで、ブログの断片を書く。
- **Tip**:
  - 機能実装を始める前に、ブログの「目的（課題感）」セクションを書いてみる。これ自体が設計の指針になる（README駆動開発に近い）。
  - **Before Implementation**: 「アプローチの選定理由（Approach Selection）」は実装前に書き切る。
    - 実装して動くか確かめるのは後でいいが、「なぜ他の案ではなくこの案を試すのか」という論理は、コードを書く前に自分の中で確定しているはずだから。

## 6. AIとの協調 (Collaborating with AI)
- **ルール**: 執筆時、AIに「足りない情報は何か？」「論理が飛躍していないか？」をレビューさせる。
- **Action**:
  - AIは、文脈から判断できない情報（設計の意図、具体的な数値など）がある場合、無理に推測せず `<!-- QUESTION: ... -->` という形式で質問を挿入する。
  - 人間は、その質問に答える形で記事を補完する。
- **Why**:
  - 「AIが足りない情報を整理し、人間がそれを補う」という分業が、質の高い記事を効率的に作る鍵だから。
